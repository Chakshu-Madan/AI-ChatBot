# StyleHub AI Support Assistant

A production-style RAG (Retrieval-Augmented Generation) customer support chatbot, built as a summer internship capstone project. It answers real product and policy questions for **StyleHub**, a demo e-commerce storefront, using locally-run embeddings and a fast open-source LLM — with no external embedding API and no vector database dependency.

**Live demo:** [style-hub-ebon.vercel.app](https://style-hub-ebon.vercel.app/) — click the chat bubble in the bottom-right corner.

**API base:** `https://ai-chatbot-227u.onrender.com`

## What it does

Ask it things like:
- *"Do you have any backpacks?"*
- *"What's the price of the men's cotton jacket?"*
- *"What's your return policy?"*
- *"Can you suggest an outfit for a birthday party?"*

The bot retrieves relevant product and FAQ data, grounds its answer in that data, and responds conversationally — while remembering the last few turns of the conversation per session.

## Architecture

```
User (browser)
    │
    ▼
React frontend (Vercel)  ──POST /chat──▶  Flask backend (Render)
                                                │
                                    ┌───────────┼───────────┐
                                    ▼           ▼           ▼
                              fastembed    NumPy cosine   Groq LLM
                              (local        similarity    (llama/mixtral
                              embeddings)   retrieval     via API)
```

**Request flow:** user message → embed query locally → retrieve top-k similar chunks from an in-memory NumPy index → build a prompt with retrieved context + recent chat history → call Groq LLM → return grounded answer + source citations.

### Key design decisions

| Choice | Why |
|---|---|
| **Local embeddings (fastembed / BAAI/bge-small-en-v1.5)** instead of an external embeddings API | Removes API rate limits and per-call cost; runs entirely on the server |
| **In-memory NumPy cosine similarity** instead of ChromaDB | ChromaDB's disk-backed queries were unreliable and occasionally hung on Render's free tier; a flat NumPy array is simpler and fast enough for a small catalog |
| **Manual retrieve → prompt → LLM flow** instead of LangChain's `ConversationalRetrievalChain` | More control over prompt construction and easier to debug when something goes wrong |
| **Index built fresh at startup**, not persisted to disk | Keeps deployment simple — no stale index files to manage; rebuild is fast enough at this scale |

## Tech stack

- **Backend:** Python, Flask, Flask-CORS, Flask-Limiter
- **Embeddings:** fastembed (`BAAI/bge-small-en-v1.5`), run locally
- **Retrieval:** NumPy (in-memory cosine similarity)
- **LLM:** Groq API
- **Frontend (demo):** React, deployed on Vercel
- **Product data:** [FakeStoreAPI](https://fakestoreapi.com/)
- **Hosting:** Render (backend), Vercel (frontend)

## API Reference

### `POST /chat`

**Request:**
```json
{
  "message": "What's your return policy?",
  "session_id": "optional-uuid-for-conversation-memory"
}
```

**Response — success (200):**
```json
{
  "answer": "We offer a 7-day return policy on clothing and accessories...",
  "sources": ["faq.txt"],
  "session_id": "generated-or-echoed-uuid"
}
```

**Response — validation error (400):**
```json
{ "error": "Message too long (max 800 characters)" }
```

**Response — rate limited (429):**
```json
{ "error": "You're sending messages too quickly. Please wait a moment and try again." }
```

**Response — server error (500):**
```json
{ "error": "Service temporarily unavailable. Please try again." }
```

### `GET /health`
Basic health check endpoint.

## Security & reliability features

- **CORS locked down** — `/chat` only accepts requests from the deployed demo frontend's origin, not any website.
- **Input validation** — rejects empty messages, non-string payloads, and messages over 800 characters.
- **Rate limiting** — 10 requests per minute per IP address (Flask-Limiter), with a friendly JSON error instead of a raw 429 page.
- **Timeout handling** — LLM calls are wrapped with a timeout so a slow/stuck request fails gracefully instead of hanging the server.

## Running locally

```bash
git clone https://github.com/Chakshu-Madan/AI-ChatBot.git
cd AI-ChatBot
pip install -r requirements.txt

# One-time: download the local embedding model
python download_model.py

# One-time: pull product data from FakeStoreAPI
python fetch_products.py

# Add your Groq API key
echo "GROQ_API_KEY=your_key_here" > .env

python app.py
```

The server starts on `http://localhost:5000` (or your configured port), builds its retrieval index from `documents/` automatically on startup, and exposes `/`, `/chat`, and `/health`.

## Known limitations

Built to be transparent about tradeoffs rather than hide them:

- **No numeric/structured filtering.** Retrieval is purely semantic similarity, not a real database query — so questions like *"anything under $20?"* can return items that don't actually meet the filter, since the model is matching on meaning, not doing arithmetic over prices.
- **Small embedding model can be swayed by literal keyword overlap.** A query like *"men's cotton t-shirt"* can sometimes retrieve a differently-named product (e.g. a jacket) that happens to share more literal words with the query than the actual best match. Rephrasing more naturally (e.g. *"do you have t-shirts?"*) generally resolves this.
- **Conversation memory is session-scoped and short.** Each browser session keeps a `session_id` in `localStorage`, and the backend retains only the last 3 exchanges as context — older parts of a conversation are forgotten, and a long-lived session can occasionally cause the bot to reference something no longer in view.
- **Free-tier hosting cold starts.** The backend spins down after ~15 minutes of inactivity; the first request afterward can take 30–50 seconds to respond.
